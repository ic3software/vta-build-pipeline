Verifiable Trust Infrastructure · Design Note

Getting Ready for Join

A credential-centric redesign of the VTA & VTC

From an ACL-and-keys world to one where a community member holds credentials, presents them, and joins — over open standards (W3C VC, SD-JWT-VC, OpenID4VCI / OpenID4VP, DCQL, DIDComm).

Shipped & merged Gated (audit / Phase-0) Planned
VTA = credential agent VTC = issuer · verifier · schema authority invite → hold → present → join
Executive summary

The shift in one sentence

We moved authorization from "a row in an ACL + a role in a JWT" to "a verifiable credential the member holds and presents" — so trust is portable, standards-native, privacy-preserving, and re-verifiable by anyone, not just by us.

Portable

Membership and roles become W3C Verifiable Credentials the holder carries — usable beyond a single VTA/VTC, by external verifiers that show up later.

Standards-native

Issuance and presentation ride OpenID4VCI / OpenID4VP + DCQL, wrapped in Trust Tasks over DIDComm. No bespoke signature schemes.

Privacy-preserving

Selective disclosure (SD-JWT-VC, BBS+), per-disclosure consent records, and a vault with no enumeration primitive.

Before  →  After

What changed architecturally

Before — ACL & keys

  • VTA = key manager (BIP-32, signing oracle) + an ACL keyspace.
  • Authorization = an ACL entry + a role claim baked into a JWT.
  • Join = bespoke: applicant submits a VP, an admin approves, the membership credential is handed back inline in the HTTP response.
  • Credentials existed (VMC / VEC) but were point-to-point artifacts — not held, not queryable, not re-presentable.
  • No wallet · no selective disclosure · no DCQL · no OID4VCI/VP.
⟶

After — hold & present

  • VTA = credential agent with a format-agnostic vault (hold) + receive / match / present.
  • Authorization = a held Role VC → a verified-assertion cache; the ACL becomes a derived index.
  • Join = invite (issue) → hold → present (DCQL) → join (verify → admit → issue), all over standards.
  • Credentials are first-class: stored encrypted-at-rest, indexed, queryable, selectively disclosed.
  • DIDComm-first transport (authcrypt = intrinsic sender auth); REST fallback.
The flow we built toward

“Getting ready for join” — the member journey

A prospective member is invited, holds the invitation, presents it on demand, and is admitted — each arrow a standards-based protocol message.

1

Invite

The VTC issues an invitation / membership credential offer to the prospect's DID — an OpenID4VCI pre-authorized offer.

VTC → holder · offer
2

Hold

The VTA redeems with a key-binding proof, receives the credential, and stores it in the vault (format-agnostic, encrypted).

holder ⇄ VTC · request → issue
3

Present

At join, the VTC verifier sends a DCQL query. The VTA matches over its vault (no enumeration), gates on consent, builds a selectively-disclosed VP.

VTC → holder → VTC · query → present
4

Join

The VTC verifies the VP (holder binding · status · issuer trust · DCQL satisfaction) → admits → issues VMC + Role VEC. The member now holds those too.

VTC · verify → admit → issue

Every step is the same credential machinery — issuance is presentation's mirror image. Build it once, reuse it for invites, memberships, roles, endorsements.

Flow 1 · Issuance

OpenID4VCI over DIDComm — offer → request → issue

VTC (issuer)VTA (holder)Vault
① credential-exchange/offer — pre-authorized code (doubles as the proof nonce)
② credential-exchange/request — holder signs an OID4VCI key-binding proof (EdDSA, bound to its did:key)
③ Issuer verifies the proof → proven holder DID must equal the credential's subject (else Forbidden)
④ credential-exchange/issue — the credential (single-use offer consumed on success only)
⑤ Holder infers format, stores via the format-agnostic vault::receive (encrypted at rest)

Holder binding is the gate

A credential is released only to a party that proves possession of the subject key — the heart of OID4VCI, and the part every implementation otherwise hand-rolls.

Relayer ≠ holder is safe

The DIDComm authcrypt sender authenticates the relayer; the inner key-binding proof authenticates the holder. The two may differ (air-gapped onboarding) with no escalation.

Single-use, DoS-resistant

The pending offer is consumed only on a successful issue — a forged or wrong-party request can't burn the legitimate holder's offer.

Flow 2 · Presentation

OpenID4VP + DCQL — query → match → present

VTC (verifier)VTA (holder)Vault
① credential-exchange/query — DCQL query + nonce + mandatory purpose (shown to the holder)
② Holder gathers candidates by the vct/type index — no wallet enumeration
③ DcqlQuery::match_credentials → which held credentials satisfy it + the claim paths to disclose
match
④ Consent gate (ISO/IEC 27560 + DPV) + selectively-disclosed present → vp_token
⑤ credential-exchange/present — holder-bound VP; verifier checks binding · status · issuer trust · DCQL

No-enumeration is a feature

The vault exposes no “list everything.” Discovery is only via an indexed field + value, so a query with no discriminator returns nothing — the holder never blind-scans its wallet to answer a verifier.

Disclose exactly what was consented

The reveal set is the consent record's personal-data list. SD-JWT-VC redacts to the consented subset; plain DI refuses to over-disclose.

Design decision · the keystone

One vault, every credential format

The vault and the exchange abstract over credential format, driven by the DCQL format selector. Adding a format never touches the storage schema — every wire op depends on this bridge.

The decisions behind it

Principles we committed to

Trust Tasks wrap OID4VCI/VP

The Trust Task is the envelope (transport · auth · threading · relayer); the body is OID4VCI / OID4VP + DCQL. We never default to "a REST endpoint + a bespoke signature scheme."

DIDComm first

Authcrypt gives intrinsic sender authentication — unpacking yields a cryptographically-proven sender DID. REST is the fallback for parties that can't speak DIDComm.

VC/VP for authorization

When the VTA attests authorization it issues a W3C VC; when a holder presents, it's a VP. No signing a JSON struct with a key and calling it auth.

Typestate for verified forms

A wire form exposes .verify() returning a distinct Verified* type. Forgetting to verify doesn't compile — it's the wrong type.

Default to DIDs

Every public-key surface is a did:key, not a raw pubkey. HPKE still works on X25519 internally — derived on demand inside the cipher layer.

Crypto-agnostic, DID-resolution-with-the-caller

The upstream key-proof layer verifies signatures via traits and surfaces the claimed key id — DID-method coupling stays out of the crate.

Security model

Where the guarantees live

Holder key-binding proof

OID4VCI openid4vci-proof+jwt: verify typ/alg, resolve the kid did:key, check the EdDSA signature (verify_strict), aud, freshness, and the c_nonce replay binding. Issuance releases only to the proven subject.

Consent-gated disclosure

A signed consent record (ISO/IEC 27560 + DPV) authorizes a verifier + claim set. The reveal set is the consent; revoked / temporally-invalid credentials are refused before any disclosure.

No-enumeration vault

There is no "list all" primitive — discovery is targeted index lookups only. A query with no discriminator yields nothing. Privacy by construction.

Sealed transfer

Every secret-bearing bundle is HPKE-sealed to a consumer client_did, ASCII-armored, with a producer assertion + out-of-band SHA-256 digest. Hardcoded suite, domain-bound info string.

Each of these was traced against forgery, replay (time + nonce), wrong-key, wrong-audience, alg-confusion, alg=none, and relayer-escalation attacks during review.

Giving back

Upstream: hardening affinidi-tdk-rs

The security-critical proof handling shouldn't be re-implemented by every consumer. We moved it upstream where SIOPv2 / OID4VCI / OID4VP all benefit.

Key-binding proof layer (openid4vci)

build_key_proof_jwt (wallet) + KeyProof::parse → verify (issuer). Crypto-agnostic: signature via the JwtSigner/JwtVerifier traits; DID resolution stays with the caller. Enforces aud, c_nonce, freshness; rejects alg=none.

Ed25519 JWS + ergonomics (oid4vc-core)

EdDsaSigner/EdDsaVerifier (RFC 8037, verify_strict) — the missing companion to the P-256 impl, since the stack is Ed25519-did:key first. Plus a string-or-array Audience helper.

Reviewed end-to-end against the bypass classes; validate_credential_request now documents that it is a structural check only — closing a real footgun where it returned Ok for a forged proof.

The road ahead

What's next

Finish presentation (3.5c / 3.4)

Consent-gate + selectively-disclosed present → vp_token; the query → present handler; the VTC verifier that emits the query and verifies the returned presentation.

Sealed & descriptors (3.6 / 3.7)

Sealed issuance to an unknown holder (the invite / air-gap case) via sealed_transfer; hand-authored Trust-Task descriptors for the family.

Role-by-VC & the cache (Phase 4)

A verified_assertions keyspace; VP-based /auth; auth extractors read the assertion record, not the JWT role. Revocation invalidates within a window.

Join ceremony integration (Phase 5)

Admit / Remint / Depart issue & revoke Role VCs and update the cache — closing the loop from "present" to "joined."

Also queued: a one-command mock VTA for test harnesses, and hierarchical (folder) trust contexts with folder-level admin.
